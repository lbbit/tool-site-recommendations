# AGENTS.md

本文件用于约束后续 AI / 自动化助手 / 协作者在维护 `tool-site-recommendations` 时的工作方式，避免把一个原本很轻的 Markdown 内容仓库改乱。

## 项目定位

这是一个以 **Markdown 内容整理与推荐** 为核心的仓库，不是复杂前端工程，也不是数据库项目。

核心目标：
- 稳定地积累工具 / 网站 / 专题推荐内容
- 保持目录结构清晰
- 保持 Pages 首页、README、文章索引长期一致
- 尽量用最轻的方式维护，不把仓库复杂化

## 工作总原则

1. **先维护内容结构，再考虑功能增强**。
2. **能简单就不要复杂**；不要为了“工程感”引入不必要框架。
3. **不要随便重构目录**；已有路径一旦公开，就应尽量稳定。
4. **每做完一项事情，都要更新 `docs-plan.md` 的状态标记。**
5. **新增任务先写进计划文档，再开始实施。**
6. **README、首页、索引数据尽量保持一致，不要三套说法。**
7. **如果改动会影响 GitHub Pages 展示，必须先自检再提交。**

## 文件与目录约定

### 允许重点维护的目录

- `articles/tools/`
- `articles/websites/`
- `articles/collections/`
- `templates/`
- `README.md`
- `index.html`
- `styles.css`
- `app.js`
- `scripts/`
- `data/`
- `.github/workflows/`
- `docs-plan.md`
- `AGENTS.md`

### 谨慎修改的内容

以下内容不要轻易大改，除非确有必要，并在计划文档中说明：

- 文章现有路径与文件命名
- `scripts/generate-index.mjs` 的输出结构
- `data/index.json` 的字段结构
- GitHub Pages 入口文件名（`index.html`）
- workflow 名称与 Pages 部署流程

### 不建议做的事

- 不要引入 React / Vue / Next.js / 数据库 / 后端服务，仅为展示文章索引
- 不要把仓库改造成复杂构建系统
- 不要无意义拆分太多配置文件
- 不要把简单问题过度工程化
- 不要为了“好看”破坏 Markdown 内容可维护性

## 新增文章规则

新增文章时应遵循：

1. 放到正确目录：
   - 工具：`articles/tools/`
   - 网站：`articles/websites/`
   - 专题：`articles/collections/`
2. 优先使用模板：`templates/recommendation-template.md`
3. 文件名使用稳定英文 slug，避免随意改名
4. 内容尽量包含：
   - 标题
   - 一句话介绍
   - 链接
   - 适合谁
   - 为什么值得推荐
   - 不足或注意点
   - 标签
5. 如果新增文章后影响索引，记得重新生成 `data/index.json`

## 修改首页 / 索引规则

如果修改首页、索引或展示逻辑，必须遵守：

1. 优先复用现有 `data/index.json` 结构
2. 如需扩展字段，先保证向后兼容
3. 改完后至少检查：
   - 首页是否能正常打开
   - 搜索是否仍可用
   - 新旧文章链接是否正常
4. 不要把简单的静态页改成依赖复杂工具链的前端应用

## 修改 README 规则

- README 是仓库的门面，改动要克制
- 不要让 README 与首页描述冲突
- 新增分类、专题、维护方式时，同步考虑是否更新 README
- 如果某个能力已完成，可在 README 中反映，但不要写与实际不符的承诺

## 计划文档更新规则

每次完成任务后，都必须：

1. 打开 `docs-plan.md`
2. 找到对应任务
3. 将状态从 `[ ]` 改成 `[x]`
4. 如有必要，在“更新记录”追加一句说明

如果新增了明显的新工作：

1. 先写入 `docs-plan.md`
2. 再开始实施

## Git 与提交规则

- 小步提交，提交信息尽量清晰
- 每次 commit 后都要 push
- 如果改动同时涉及内容、脚本、首页，提交信息应能看出主题
- 不要把临时文件、抓包文件、调试产物提交进仓库

## 推荐的工作顺序

处理任务时，优先按这个顺序：

1. 阅读 `README.md`
2. 阅读 `docs-plan.md`
3. 判断本次是“内容补充”还是“展示/自动化增强”
4. 实施改动
5. 如有索引相关变更，运行 `node scripts/generate-index.mjs`
6. 检查 `docs-plan.md` 是否需要标记完成
7. 提交并 push

## 维护目标

长期目标不是把仓库做得越来越复杂，而是：

- 内容越来越丰富
- 结构越来越稳定
- 首页越来越好用
- 维护成本越来越低

如果某项改动会提升复杂度，但不能明显提升内容维护效率，优先不要做。
